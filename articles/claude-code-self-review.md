Claude Code で実装して PR を出すまでのワークフローで個人的に工夫したことを紹介します。

## 問題：PR 後の AI レビュー対応にかかるコスト

Claude Code に実装を任せて PR を出すと、GitHub Copilot からレビューが来ます。このレビュー自体の質は高くて、本当に危ないバグを見つけてくれることもあります。

ただ、面倒なのはその仕分けです。大量に来る指摘を読んで、対応すべきか無視して良いかを判断しないといけません。実際のところ、無視して良い指摘もかなり多いです。

- 確かにそうだけど、今回は最小限の修正にしたいから大きく変えたくない
- 他のコードとの統一感を考えるとこのままでいい
- 実際にそこにその値が来ることはない

こういった「文脈を踏まえれば対応不要」な指摘を、PR を出した後に非同期で受け取って、読んで、仕分けして、必要なものだけ直して、またプッシュして…という往復コストが積み重なります。

どうせ AI がレビューするなら、**PR を出す前に言ってほしい**。その場で直せるものは直して、対応しないものは理由を添えて PR に書いておけば、後から同じ指摘が来ることもなくなります。

そこで、Claude Code が PR を出す前に別のエージェントにレビューさせる仕組みを作りました。

## なぜ「別の」エージェントか

実装した本人に「レビューして」と頼んでも、自分が書いたコードに対するバイアスがあって問題に気づきにくいです。これは人間でも AI でも同じで、自分の成果物を評価させると甘くなりがちです。なので実装者と評価者を分けてみることにしました。（[Anthropic のブログ記事](https://www.anthropic.com/engineering/harness-design-long-running-apps)でも実装者と評価者の分離が推奨されています。）

Claude Code の Agent ツール（subagent）は親の会話コンテクストを持たないので、差分だけ渡せばフレッシュな目でレビューできます。

## 仕組み

やっていることはシンプルです。

1. Claude Code のスラッシュコマンド（`/self-review`）を実行する
2. master からの差分とコミット履歴を収集する
3. それを subagent に渡してレビューさせる
4. レビュー結果を受け取った親エージェントが判断して対応する

## スラッシュコマンドの実装

`~/.claude/commands/self-review.md` に個人用のスラッシュコマンドを作りました。リポジトリには入れず、あくまで個人の設定です。

```markdown
---
description: PR 前のセルフレビュー。別エージェントが差分をレビューし、明らかなバグは自動修正する
allowed-tools: Bash(git diff:*), Bash(git log:*), Bash(git merge-base:*), Agent, Read, Edit, Write, Glob, Grep
---

# Pre-PR Self-Review

## Context

- 差分: !`git diff $(git merge-base HEAD master)..HEAD`
- コミット履歴: !`git log --oneline $(git merge-base HEAD master)..HEAD`
- 未コミットの変更: !`git diff HEAD`
- 追加指示: $ARGUMENTS

## Instructions

Agent ツールを起動し、上記の差分・コミット履歴をそのまま渡して自由にレビューさせる。
追加指示があればそれも渡す。

レビュー結果を受け取ったら各指摘を自分で判断し対応する:
- 修正すべき → 自動修正
- 直さなくていい → 無視
- コードだけで判断できない → ユーザーに確認

最後にユーザーへ日本語で報告:
- 修正した項目、無視した項目（理由付き）、確認待ちの項目を全て見せる

### PR description 用

対応しなかった指摘とその理由を、後で PR 作成時に description へ組み込めるよう記憶しておく。
形式例: `## Self-review\n- xx は yy のため対応不要`
```

### Context セクション

`!` 記法を使って、コマンド実行時に差分やコミット履歴を自動で収集しています。これがそのまま subagent に渡されます。`$ARGUMENTS` で「認証周りを重点的に見て」のような追加指示も渡せます。

## やってみた結果

実際に GitHub Actions のワークフロー定義を修正する PR で試してみました。

![/self-review の実行結果](/images/claude-code-self-review/self-review.png)

subagent が YAML のインデント混入を見つけて、親エージェントがその場で自動修正しています。対応不要と判断した指摘は理由付きでテーブルにまとめて報告してくれました。こういった問題を PR 前に潰せるのは助かります。

## 設計のポイント

### レビュアーには指摘を全部出させる

仕分けが大変とは言いましたが、レビュアー側で指摘を絞ると本当に必要だった指摘まで落ちてしまいます。何を無視するかはレビューを受けた側が決めればいいので、レビュアーには気になったことを全部出させています。

### 指摘への対応は親エージェントが判断する

subagent のレビュー結果を受け取った親エージェントが、文脈を踏まえて修正するか無視するかを判断します。コードだけでは判断できないものはユーザーに確認します。無視した指摘もユーザーには見せるようにしていて、親エージェントの判断が間違っていた場合に人間が気づけるようにしています。

### PR description に「対応しなかった指摘」を書く

レビューで指摘されたけど意図的に対応しなかった項目は、PR の description に書くようにしています。「検討済みだけど対応しない」ことが明記されていれば、Copilot や人間のレビュアーが同じ指摘を繰り返さずに済むのではないか、という狙いです。まだ検証はできていないので、効果があるかは今後試していきます。

### `/self-review` の実行忘れを防ぐ

Claude Code の hooks 機能で、`gh pr create` の実行時に `/self-review` 済みか確認する prompt hook も設定しています。ただし、正直なところ動作は気まぐれで、確実に動く仕組みにはまだなっていません。

## まとめ

PR 後に来るレビューの仕分けが面倒なら、PR 前にやってしまえばいいのでは、という発想で作ってみました。実装したエージェントとは別のエージェントにレビューさせて、その結果を受けて修正するところまでを PR 前に済ませます。ちなみにこの記事自体も、書いたエージェントとは別のエージェントにレビューしてもらっています。まだ荒削りなところもありますが、何かの参考になれば幸いです。
