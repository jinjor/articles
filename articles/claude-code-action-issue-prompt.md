## Claude Code Action で Pull Request にチェックリストを投稿する

[Claude Code Action](https://github.com/anthropics/claude-code-action) を使うと、GitHub Actions で Claude Code を動作させて Pull Request のレビューなどをさせることができます。
この記事では細かい使い方は紹介しません。なぜなら Claude Code に頼めばすぐにワークフロー定義を YAML で書いてくれるからです（笑）

依頼している内容自体は本題ではないのですが、簡単に何をしたのかということを説明します。
やったことは、Pull Request に対して「AI によるレビューがしにくい部分」を「人間が確認する」ためにチェックリストを投稿することです。例えば、以下のような内容です。

- ローカルで UI の動作確認をしたか（人間が目で見て挙動を確認する必要がある）
- リストは正しい順序で表示されているか（考慮しなければいけないが「正解」がない）
- 長い文字列が来ても適切に表示できるか（パターンが多く判定方法を指示しにくい）
- Figma のデザイン通りに実装できているか（仕組みの整備が追いついていない）

![PR に投稿されたチェックリストの例](/images/claude-code-action-issue-prompt/checklist.png)

チェックリストを投稿するためにわざわざ Claude Code を使っているのは、以下のような理由からです。

- PR の内容に応じてピンポイントでチェック項目を出したい（関係ない項目が大量に出るとそのうち見なくなる）
- チェック項目を提示する条件をどう形式化するかを考えるのがコスト

この辺を Claude Code はよしなにやってくれるので、とても助かっています。

## プロンプトを GitHub Issue に切り出す

ここからが本題なのですが、ここでの工夫として **Issue の description に書いたプロンプトを CI から読んで使う**ことにしました。

![Issue の description にプロンプトを記述した例](/images/claude-code-action-issue-prompt/issue-prompt.png)

このようにしたメリットはいくつかあります。

### Issue の内容を編集するだけでプロンプトの更新が可能

プロンプトを最適化するためには試行錯誤が必要ですが、修正するたびにコードをプッシュ・PR をマージして反映するのは苦痛です。プロンプトを Issue に切り出すことで、**Issue の内容を編集するだけで新しいプロンプトを試せる**ようになります。
これにより、**コードベースから解放されることで変更のコストを下げ、改善サイクルを高速に回す**ことができるようになりました。

### Issue の Assignee で購読者を管理

ここで紹介しているチェックリストの投稿は、**Issue の Assignee に登録された人のみ適用される**ようになっています。このチェックリストが気に入れば Assignee に自分を登録してください、ということです。

ナレッジワークではプロダクトのコードベースをモノレポで管理しています。これは情報が１箇所に集約されるという点でメリットがありますが、複数のチームの事情が混在しやすいというデメリットもあります。
元々このチェックリストの仕組みは、所属チームで「考慮漏れによるミス」が頻発したことから考案されました。しかし、全体に適用すると他チームと合意を取る必要が生まれて素早く動けなくなる懸念がありました。

**提供を希望する人が opt-in で購読する**仕組みによって、この問題を解決することができました。また、購読するメンバーを変更するのにコードを変更する必要もありません。

### 動的に新しいプロンプトの追加が可能

プロンプトを記述した Issue は決め打ちではなく、特定のラベルがついているものを取得するようになっています。つまり、**Issue を増やしてラベルをつければ新しいプロンプトを適用できる**ということになります。
小さく試しているのでまだ数は 2 つしかないのですが、理屈としてはいくらでも増やせます。プロンプトの内容は自由なので、実はチェックリスト以外のレビューも普通にさせることができます。

## その他の工夫

### 全てを Claude Code にやらせない

Claude Code （に限らず、おそらく他の AI エージェントも）はかなり気まぐれに振る舞います。最初、コメントの投稿まで任せていましたが、なぜか「投稿しなくて良いだろう」と判断して投稿してくれませんでした。これでは困るので Claude Code に任せるのはコメントの内容を出力するところまでにして、投稿はスクリプトに切り出しました。

同様の理由で、購読者を絞るところも事前のスクリプトでやっています。これは無駄に Claude Code を起動しないことでコストを節約するためでもあります。

**確実に遂行したいことはスクリプトを実行し、柔軟にやってほしいことは AI にお願いする**という境界を明確に意識することが重要だと思いました。

### 判断理由を答えさせる

時々チェック項目の中に「この PR の内容から、なぜその項目をチョイスした？」と思えるものが紛れ込むことがありました。判断理由を追いやすくするために、HTML コメント（`<!-- ... -->`）で**判断理由を書かせる**ようにしました。コメントを編集状態にすると、なぜそのチェック項目が選択されたのかが確認できます。

![コメントを編集状態にすると判断理由が確認できる](/images/claude-code-action-issue-prompt/show-reason.png)
